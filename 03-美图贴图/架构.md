## 5. 核心算法：工业版透明图中轴线

之前失败/偏左的原因：旧算法用有效像素左右边界框中心：

```python
cx = (xs.min() + xs.max()) / 2
```

这个算法会被不规则透明图、右侧细小虚边、散点、半透明边缘拉偏，导致视觉主体偏左或偏右。

成功版本改为 Alpha 加权重心：

```python
from PIL import Image as _I
import numpy as _N
ALPHA_THRESHOLD = 20

def get_off(p):
    try:
        img = _I.open(p).convert("RGBA")
        a = _N.array(img)
        h = a.shape[0]
        alpha = a[:, :, 3].astype(_N.float64)
        mask = alpha >= ALPHA_THRESHOLD
        if not mask.any():
            return 0, 0

        ys, xs = _N.where(mask)
        weights = _N.where(mask, alpha, 0.0)
        y_min = int(ys.min())

        # 工业版透明图中轴线：Alpha>=20 的透明度加权重心
        cx = float((_N.indices(alpha.shape)[1] * weights).sum() / weights.sum())

        s = 360.0 / h
        dy = int(round(TARGET_Y - y_min * s - DEFAULT_TOP))
        dx = int(round(TARGET_X - cx * s - DEFAULT_LEFT))
        return dx, dy
    except:
        return 0, 0
```

当前正式脚本为了保持极简，`DEFAULT_TOP` / `DEFAULT_LEFT` 在公式里仍写成 590 / 1921；语义上等价。

关键点：

- 有效像素：原始 Alpha >= 20
- 中轴线：`cx = sum(x * alpha) / sum(alpha)`
- 只统计有效像素，低于 20 的透明噪点不参与
- Y 方向仍用最高有效像素 `y_min`，保证顶端不高于 TARGET_Y=620
- `TARGET_X` / `TARGET_Y` 必须用变量，不允许在 `get_off()` 里硬编码

DS0046 校准结果：

```text

```

### 问题 3：中轴线偏左

现象：

旧算法按 `(xs.min()+xs.max())/2` 对齐，数学上对了，但视觉上贴图整体偏左。

原因：

- 不规则透明图的左右边界框中心不等于视觉重心
- 细小虚边/散点会拉大边界框

解决：

使用 Alpha>=20 的透明度加权重心：

```python
weights = np.where(mask, alpha, 0.0)
cx = (x_indices * weights).sum() / weights.sum()
```

### 问题 4：TARGET_X 改了但实际没生效

现象：

顶部 `TARGET_X=2110`，但 `Offset` 仍按 2105 计算。

原因：

`get_off()` 内部硬编码了：

```python
dx = round(2105 - cx*s - 1921)
dy = round(620 - y_min*s - 590)
```

解决：

改成变量：

```python
dx = round(TARGET_X - cx*s - DEFAULT_LEFT)
dy = round(TARGET_Y - y_min*s - DEFAULT_TOP)
```

正式脚本目前 dx/dy 使用变量 TARGET_X/TARGET_Y，但 1921/590 仍为固定常量；如果以后重构，必须保持语义不变。

### 问题 8（v3.2）：旋转手柄坐标偏移导致贴图不旋转

现象：

贴图没有旋转步骤，直接就被拖动定位了。结果贴图角度不对。

原因：

`coords.json` 中旋转手柄坐标标定有误：
- `rotate_btn: [822, 975]` — 实际旋转手柄在 y=994，差了 19px
- `rotate_drop: [827, 975]` — 同样 y 轴差 19px

标定时的窗口分辨率是 1280x1440，但后来在 1280×实际屏幕高度（1920x1080 屏幕上的 1440？）使用，或者美图版本更新后旋转手柄位置发生了变化。鼠标按下去的位置不在旋转手柄上，所以那 5px 拖动在画布空白处什么都没发生，代码继续执行到定位拖动步骤，直接拖贴图到偏移位置，造成了「没有旋转，直接拖动上下」的效果。

解决：

重新标定旋转手柄坐标。让爸爸把鼠标悬停在旋转手柄上，读取屏幕坐标：

```
屏幕坐标: (2102, 994)  →  窗口内相对: (822, 994)
```

更新 coords.json：
- `rotate_btn: [822, 975]` → `[822, 994]`（y 轴 +19px）
- `rotate_drop: [827, 975]` → `[827, 994]`（y 轴 +19px）

修复后测试：49/51 全部成功，旋转正常。

教训：

1. 桌面 GUI 自动化中，按钮/手柄的像素级坐标是**脆弱的**——美图版本更新、窗口分辨率变化、屏幕 DPI 变化都可能导致偏移
2. 标定坐标时，应该用**当前实际窗口尺寸**来验证，不能依赖历史标定值
3. 以后如果旋转又失效，第一步就是让爸爸把鼠标移到旋转手柄上读坐标
4. 考虑以后增加坐标校验步骤：窗口实际尺寸与参考尺寸偏差超过阈值时报警
